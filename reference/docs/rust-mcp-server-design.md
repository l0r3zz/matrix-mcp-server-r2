# Rust Matrix MCP Server -- Design Document

## 1. What This Document Covers

This document explains the design decisions, patterns, and final code structure
of **matrix-mcp-server-r2**, a Rust rewrite of the original TypeScript
`matrix-mcp-server`. It is intended for someone with beginner-to-intermediate
Rust experience who wants to understand *why* the code is shaped the way it is.

If you are new to the project, start here. If you just want to run the server,
see the top-level `README.md`.

---

## 2. Why Rust?

The TypeScript version works. Porting it to Rust is motivated by two things:

| Goal | How Rust helps |
|------|----------------|
| **Speed** | Native binary; no V8 startup, no JIT warm-up. Sub-millisecond tool dispatch. |
| **Robustness** | Compile-time guarantees against null dereferences, data races, and use-after-free. The `Result<T, E>` type forces you to handle errors. |

Two future goals also drove the choice:

- **E2EE support** -- The `matrix-sdk-crypto` crate provides Olm/Megolm
  encryption in native Rust. No FFI bridge is needed.
- **Single-binary deployment** -- One ~10 MB static binary replaces Node.js +
  `node_modules/`. Easier to put inside a Docker container alongside Agent Zero.

---

## 3. High-Level Architecture

```
                           HTTP
  MCP Client  ─────────>  axum  ────>  rmcp StreamableHttpService
  (Cursor, A0)  POST /mcp                       │
                                          MatrixMcpServer
                                          (20 #[tool] methods)
                                                 │
                                           matrix-sdk Client
                                                 │
                                          Matrix Homeserver
```

Three crates carry most of the weight:

| Crate | Role |
|-------|------|
| **rmcp** (v1.2) | The official Rust MCP SDK. Provides protocol framing, tool registration macros, and the Streamable HTTP transport. |
| **matrix-sdk** (v0.10) | High-level Matrix client. Handles sync, room state, message sending, and (eventually) E2EE. |
| **axum** (v0.8) | HTTP server framework. Mounts the rmcp service at `/mcp` and serves a `/health` endpoint. Uses Tower middleware for CORS and tracing. |

---

## 4. Crate Selection -- What We Tried and Why

### 4a. MCP Protocol Layer: rmcp

The original TypeScript server uses the `@modelcontextprotocol/sdk` npm package
and its `StreamableHTTPServerTransport`. For the Rust port we needed an
equivalent.

**Options considered:**

1. **Hand-rolled JSON-RPC** -- Build the MCP framing ourselves with axum routes.
   This was the approach in the Phase-0 scaffolding (custom `Tool` trait, SSE
   endpoints). It compiled, but it did not speak the MCP protocol correctly and
   would have required ongoing maintenance as the spec evolved.

2. **rmcp** (official Rust MCP SDK) -- Provides `#[tool]` / `#[tool_router]` /
   `#[tool_handler]` proc macros, a Tower-compatible `StreamableHttpService`,
   and session management out of the box.

We chose **rmcp**. The macros eliminate boilerplate, the transport is
protocol-correct, and the crate is maintained by the MCP spec authors.

### 4b. Matrix Client: matrix-sdk

The TypeScript version uses `matrix-js-sdk`. The Rust equivalent is
`matrix-sdk`, which is built on top of the `ruma` crate (typed Matrix API
bindings).

We use `matrix-sdk` with only the `rustls-tls` feature enabled. The
`e2e-encryption` feature is available behind a Cargo feature flag (`e2ee`) but
is not compiled by default for v1.

### 4c. HTTP Layer: axum + tower-http

Axum was already in the Phase-0 scaffolding and is the natural partner for rmcp
because both are built on Tower. The `tower-http` crate adds CORS and request
tracing as middleware layers.

## 5. The rmcp Macro System -- How Tools Are Defined

This section walks through the macros that turn plain Rust methods into MCP
tools. If you have not used proc macros before, think of them as compile-time
code generators: you annotate your code, and the macro writes extra code for you.

### 5a. The Server Struct

```rust
#[derive(Clone)]
pub struct MatrixMcpServer {
    client: Client,                  // matrix-sdk Client (Clone is cheap -- it wraps an Arc)
    config: Arc<Config>,             // shared, read-only config
    tool_router: ToolRouter<Self>,   // generated by the macro below
}
```

`ToolRouter<Self>` is a lookup table from tool name to handler function. The
`#[tool_router]` macro generates it.

### 5b. `#[tool_router]` -- Registering Tools

```rust
#[tool_router]                             // <-- proc macro on the impl block
impl MatrixMcpServer {
    #[tool(description = "List joined rooms")]   // <-- marks this method as an MCP tool
    async fn list_joined_rooms(&self) -> Result<CallToolResult, McpError> {
        // ...
    }
}
```

What the macro does at compile time:

1. Scans every method marked `#[tool(description = "...")]`.
2. Generates a `fn tool_router() -> ToolRouter<Self>` associated function that
   maps tool names to handlers.
3. The tool **name** defaults to the method name with underscores (e.g.,
   `list_joined_rooms`). The MCP client sees this name.
4. The tool **description** becomes the `description` field in the MCP
   `tools/list` response.

### 5c. `#[tool_handler]` -- Wiring into ServerHandler

```rust
#[tool_handler]
impl ServerHandler for MatrixMcpServer {
    fn get_info(&self) -> ServerInfo { /* ... */ }
}
```

`ServerHandler` is the trait that rmcp requires every MCP server to implement.
The `#[tool_handler]` macro auto-generates the `call_tool` and `list_tools`
methods by delegating to the `ToolRouter` that `#[tool_router]` built. You only
write `get_info` yourself.

### 5d. Tool Parameters -- The `Parameters<T>` Wrapper

When a tool needs input (e.g., a room ID), you pass a struct wrapped in
`Parameters<T>`:

```rust
#[derive(Debug, Deserialize, schemars::JsonSchema)]
pub struct RoomIdInput {
    #[serde(rename = "roomId")]
    pub room_id: String,
}

#[tool(description = "Get room info")]
async fn get_room_info(
    &self,
    Parameters(input): Parameters<RoomIdInput>,
) -> Result<CallToolResult, McpError> {
    let room = self.get_room(&input.room_id)?;
    // ...
}
```

How this works:

- `Parameters<T>` is an rmcp extractor. When an MCP client calls the tool with
  `{"roomId": "!abc:example.com"}`, rmcp deserializes that JSON into
  `RoomIdInput` and hands you the populated struct.
- The `schemars::JsonSchema` derive generates the JSON Schema that appears in
  `tools/list`. This is how MCP clients (Cursor, Claude, etc.) learn what
  parameters the tool accepts.
- `#[serde(rename = "roomId")]` maps the Rust field `room_id` (snake_case) to
  the JSON property `roomId` (camelCase) expected by MCP clients.

**Why a struct instead of individual parameters?**

rmcp lets you list parameters directly on the method signature, but those
parameters use the Rust variable name as the JSON key. Since the TypeScript API
uses camelCase (`roomId`, `startDate`, `messageType`), we would need
non-idiomatic Rust names. Using a struct with `#[serde(rename)]` keeps the Rust
code in snake_case while the JSON stays in camelCase.

### 5e. Tool Return Values

Every tool returns `Result<CallToolResult, McpError>`:

- `Ok(CallToolResult::success(vec![Content::text("...")]))` -- success with text content.
- `Ok(CallToolResult::error(vec![Content::text("Error: ...")]))` -- a **tool-level** error
  (the MCP call succeeded, but the operation failed). The `is_error` flag is `true`.
- `Err(McpError { ... })` -- an **MCP protocol error** (e.g., unknown tool).

The distinction matters: `Ok(err_text("Room not found"))` tells the AI "the tool
ran but the room doesn't exist." `Err(...)` tells the MCP framework "something
is wrong with the protocol."

We have three helper functions to reduce boilerplate:

```rust
fn single_text(text: impl Into<String>) -> CallToolResult   // one text block, success
fn text_result(texts: Vec<String>) -> CallToolResult          // many text blocks, success
fn err_text(text: impl Into<String>) -> CallToolResult        // one text block, is_error = true
```

## 6. Matrix SDK Integration

### 6a. Client Creation and Session Restore

The TypeScript server calls `sdk.createClient()` then `client.setAccessToken()`.
In Rust, the equivalent is:

```rust
let client = Client::builder()
    .homeserver_url(homeserver_url)
    .build()
    .await?;

let session = MatrixSession {
    meta: SessionMeta { user_id, device_id: OwnedDeviceId::from("MCP_SERVER") },
    tokens: MatrixSessionTokens { access_token, refresh_token: None },
};
client.matrix_auth().restore_session(session).await?;
```

Key points:

- `Client::builder()` is the entry point. It returns a `ClientBuilder`.
- `restore_session` replays an existing access token instead of doing a full
  login. This matches the TS pattern where the token comes from env vars.
- We use a fixed device ID (`"MCP_SERVER"`) so the homeserver always sees the
  same device.

### 6b. Initial Sync and Background Sync

Before tools can read rooms, the client must sync:

```rust
client.sync_once(SyncSettings::default().timeout(Duration::from_secs(30))).await?;
```

After the initial sync, a background task keeps state fresh:

```rust
tokio::spawn(async move {
    loop {
        client.sync_once(settings.clone()).await;
        tokio::time::sleep(Duration::from_secs(1)).await;
    }
});
```

This mirrors the TS `client.startClient({ initialSyncLimit: 100 })`.

### 6c. Client Cache

The `ClientCache` struct stores `matrix_sdk::Client` instances keyed by
`(user_id, homeserver_url)` with a 15-minute TTL. A background task runs every
5 minutes and evicts expired entries.

```rust
pub struct ClientCache {
    entries: Arc<RwLock<HashMap<String, CachedClient>>>,
}
```

`Arc<RwLock<...>>` is the standard Rust pattern for shared mutable state across
async tasks:

- `Arc` (atomic reference counting) lets multiple tasks hold a pointer to the
  same data.
- `RwLock` (from tokio) allows many concurrent readers **or** one exclusive
  writer.

### 6d. Reading Timeline Events

`matrix-sdk` 0.10 returns `TimelineEvent` objects from `room.messages()`. Each
event has a `kind` field (a `TimelineEventKind` enum) rather than a simple JSON
blob. To extract sender, body, and timestamp, we serialize the `kind` to JSON
and parse it:

```rust
fn parse_message_events(chunk: &[TimelineEvent]) -> Vec<(String, String, i64)> {
    for event in chunk {
        let json_str = serde_json::to_string(&event.kind).unwrap_or_default();
        if let Ok(val) = serde_json::from_str::<serde_json::Value>(&json_str) {
            // extract sender, body, origin_server_ts
        }
    }
}
```

This is a pragmatic choice: the typed event API in matrix-sdk is powerful but
deeply nested. For a port focused on API parity with the TypeScript version,
working with raw JSON is simpler and matches how the TS code uses
`event.getContent().body`.

## 7. Transport -- Connecting rmcp to axum

The rmcp crate provides `StreamableHttpService`, a Tower `Service` that speaks
the MCP Streamable HTTP protocol. It is mounted in axum using `nest_service`:

```rust
let mcp_service: StreamableHttpService<MatrixMcpServer> = StreamableHttpService::new(
    move || Ok(MatrixMcpServer::new(client.clone(), config.clone())),  // factory
    LocalSessionManager::default().into(),                              // session mgr
    StreamableHttpServerConfig::default(),                              // config
);

let app = Router::new()
    .route("/health", get(health_handler))
    .nest_service("/mcp", mcp_service);
```

Three arguments to `StreamableHttpService::new`:

1. **Factory function** `|| Ok(server)` -- called once per MCP session. Each
   session gets its own `MatrixMcpServer` instance, but they all share the same
   `Client` (which is cheap to clone because it wraps an `Arc`).
2. **Session manager** -- `LocalSessionManager` stores sessions in memory.
3. **Config** -- default settings (stateful mode with session IDs).

The factory returns `Result` so you can fail session creation if something is
wrong. We always return `Ok`.

## 8. Error Handling Strategy

Rust does not have exceptions. Instead, functions return `Result<T, E>`. This
project uses two error enums:

### `AppError` (application-level)

Defined in `src/error.rs` with `thiserror::Error`. Used for startup failures,
config errors, and Matrix SDK errors. The `#[from]` attribute auto-generates
`From<matrix_sdk::Error> for AppError`, so you can use `?` to propagate errors.

```rust
#[derive(Error, Debug)]
pub enum AppError {
    #[error("Configuration error: {0}")]
    Config(String),

    #[error("Matrix SDK error: {0}")]
    MatrixSdk(#[from] matrix_sdk::Error),

    // ...
}
```

### `McpError` (tool-level)

This is `rmcp::ErrorData`, aliased as `McpError`. Returned by tool methods when
something goes wrong at the protocol level. In practice, most tool errors are
returned as `Ok(err_text("..."))` rather than `Err(McpError)`, because the MCP
spec distinguishes between "the tool worked but the operation failed" and "the
MCP call itself failed."

## 9. Configuration

All configuration comes from environment variables (loaded by `dotenvy` from a
`.env` file if present). The `Config` struct has three required fields:

```
MATRIX_HOMESERVER_URL=https://matrix.example.com
MATRIX_USER_ID=@bot:example.com
MATRIX_ACCESS_TOKEN=syt_...
```

Everything else is optional and has sensible defaults. The full list is in
`.env.example`.

The `validate()` method checks that the homeserver URL starts with `http://` or
`https://` and that the user ID has the `@user:domain` format. These are cheap
sanity checks that catch typos before the server tries to connect.

## 11. Key Patterns and Idioms

### 11a. The "get_room-or-error" Helper

Most tools start by looking up a room. Rather than repeat the same error
handling 15 times, we have:

```rust
fn get_room(&self, room_id: &str) -> std::result::Result<Room, CallToolResult> {
    // try to parse, try to look up, return Err(err_text(...)) on failure
}
```

Callers use it like this:

```rust
let room = match self.get_room(&input.room_id) {
    Ok(r) => r,
    Err(e) => return Ok(e),   // e is already a CallToolResult with is_error = true
};
```

Note the `return Ok(e)` -- we are returning a *successful* MCP response that
contains an error message for the AI.

### 11b. Async Iteration Instead of Closures

Some `matrix-sdk` methods like `get_member_no_sync()` are `async`. You cannot
call `.await` inside a closure passed to `.filter()` or `.find()`. The solution
is a plain `for` loop:

```rust
// Cannot do: rooms.iter().find(|r| r.get_member_no_sync(&uid).await.is_some())

let mut existing_dm = None;
for room in self.client.joined_rooms() {
    if let Ok(Some(_)) = room.get_member_no_sync(&target_uid).await {
        existing_dm = Some(room);
        break;
    }
}
```

This is a common pattern in async Rust -- iterators and closures do not support
`.await`, so you fall back to imperative loops.

### 11c. `Arc` for Shared Ownership

The `Config` is created once and read by every tool. Instead of cloning the
entire struct into each server instance, we wrap it in `Arc<Config>`:

```rust
let config = Arc::new(config);
// ...later, in the factory...
move || Ok(MatrixMcpServer::new(client.clone(), config.clone()))
```

`Arc::clone()` increments a reference count (cost: one atomic CPU instruction).
It does **not** copy the config data. This is idiomatic Rust for sharing
immutable data across tasks.

### 11d. `serde` + `schemars` for JSON Schema Generation

Each input struct derives both `serde::Deserialize` (for JSON parsing) and
`schemars::JsonSchema` (for JSON Schema generation). The rmcp framework uses the
schema to tell MCP clients what parameters each tool accepts.

```rust
#[derive(Debug, Deserialize, schemars::JsonSchema)]
pub struct CreateRoomInput {
    #[serde(rename = "roomName")]
    pub room_name: String,

    #[serde(rename = "isPrivate")]
    pub is_private: Option<bool>,

    pub topic: Option<String>,
    // ...
}
```

Fields marked `Option<T>` are optional in the JSON schema. Required fields are
plain `T`.

## 12. File Layout

```
src/
  main.rs               Entry point. Wires config, matrix client, rmcp service, axum, and shutdown.
  lib.rs                Module declarations and re-exports for the library target.
  config.rs             Config struct loaded from env vars via dotenvy.
  error.rs              AppError enum with thiserror derives.
  auth.rs               MatrixAuthContext struct (placeholder for per-request auth).
  matrix/
    mod.rs              Re-exports create_matrix_client and ClientCache.
    client.rs           matrix-sdk client creation, session restore, background sync.
    cache.rs            TTL-based in-memory client cache with cleanup task.
  mcp/
    mod.rs              Re-exports server module.
    server.rs           MatrixMcpServer struct, all 20 tool methods, input structs, ServerHandler impl.
tests/
  config_tests.rs       Validation logic (URL format, user ID format).
  cache_tests.rs        Cache miss, shutdown behavior.
  server_tool_list_tests.rs  Compile-time check that ServerHandler is implemented.
```

### Why One Big `server.rs`?

The rmcp `#[tool_router]` macro processes a single `impl` block. All `#[tool]`
methods must be in that block. This naturally leads to one file containing all
20 tools plus their input structs.

You *could* split the input structs into a separate module, but keeping them
next to the methods that use them makes the code easier to navigate. At ~680
lines the file is large but manageable.

## 13. Lifecycle of a Tool Call

Here is the full path of an MCP `tools/call` request:

1. **MCP client** sends `POST /mcp` with JSON-RPC body:
   `{"method": "tools/call", "params": {"name": "get_room_info", "arguments": {"roomId": "!abc:example.com"}}}`.

2. **axum** receives the HTTP request, applies CORS and tracing layers, and
   forwards it to the nested `StreamableHttpService`.

3. **rmcp** deserializes the JSON-RPC envelope, looks up the session (or creates
   one), and dispatches to `MatrixMcpServer::call_tool()` (auto-generated by
   `#[tool_handler]`).

4. **ToolRouter** finds the handler for `"get_room_info"` and calls it, passing
   the deserialized `RoomIdInput`.

5. **get_room_info** calls `self.get_room(room_id)`, which delegates to
   `self.client.get_room(owned_id)` from `matrix-sdk`.

6. The tool builds a `CallToolResult::success(vec![Content::text("Room Information:\n...")])`.

7. **rmcp** serializes the result back to JSON-RPC and returns it to axum.

8. **axum** sends the HTTP response.

## 14. Graceful Shutdown

The server listens for `ctrl_c` (which also covers `SIGTERM` in containers):

```rust
axum::serve(listener, app)
    .with_graceful_shutdown(async move {
        let _ = tokio::signal::ctrl_c().await;
        cache_for_shutdown.shutdown_all().await;   // drop cached Matrix clients
        sync_handle.abort();                        // stop background sync
        cleanup_handle.abort();                     // stop cache cleanup
    })
    .await?;
```

This ensures in-flight requests finish before the process exits, and the Matrix
clients are cleaned up.

## 15. What Is Not Yet Implemented

| Feature | Status | Notes |
|---------|--------|-------|
| **E2EE** | Feature-flagged (`--features e2ee`) | `matrix-sdk-crypto` is in the dependency tree but no tools use it yet. The v2 spec defines `-r2` tools for encrypted rooms. |
| **OAuth / Token Exchange** | Config fields exist | The TS server supports optional OAuth with Keycloak. The config struct has the fields; the wiring is not done. |
| **Per-request auth headers** | Struct defined in `auth.rs` | The TS server extracts `matrix_user_id` / `matrix_access_token` from HTTP headers. The Rust version currently uses server-wide env vars. The `MatrixAuthContext` struct is ready; the middleware is not wired. |
| **Reply-to for messages** | Parameter accepted, not implemented | `send_message` accepts `replyToEventId` but does not set `m.relates_to` on the event content. |
| **Integration tests** | Scaffolding only | `wiremock` is in dev-deps. Real tests against a mock homeserver are a next step. |

## 16. Lessons Learned During the Port

### The rmcp `#[tool]` attribute resolution issue

Inside `#[tool_router] impl { ... }`, the inner `#[tool(...)]` attributes are
**helper attributes** consumed by the proc macro. However, `rmcp::tool` is
*also* exported as a standalone proc-macro attribute. If you write
`use rmcp::tool;`, the Rust compiler tries to resolve `#[tool(aggr)]` on
parameters as a proc-macro invocation (which is not allowed on parameters). The
fix: import `tool` for method-level use, and use `Parameters<T>` for struct
extraction instead of `#[tool(aggr)]`.

### `matrix-sdk` session API path

In `matrix-sdk` 0.10, the session types live at
`matrix_sdk::authentication::matrix::{MatrixSession, MatrixSessionTokens}`, not
the more intuitive `matrix_sdk::matrix_auth::*`. The `Client::restore_session`
method does not exist directly on `Client`; you go through
`client.matrix_auth().restore_session(session)`.

### Non-exhaustive structs in rmcp

`ServerInfo` is marked `#[non_exhaustive]`, which means you cannot construct it
with struct literal syntax from outside the rmcp crate. The solution is to use
the provided constructor and builder methods:

```rust
ServerInfo::new(capabilities)
    .with_server_info(Implementation::new("name", "version"))
    .with_instructions("...".to_string())
```

### `TimelineEvent` in matrix-sdk 0.10

`TimelineEvent` does not have a `.event` field. It has `.kind` (a
`TimelineEventKind` enum). To get raw JSON, we serialize `.kind` with
`serde_json::to_string()` and parse the result. This is less elegant than the
TypeScript `event.getContent().body` but works reliably across all event types.

### Async closures do not exist (yet)

Rust closures cannot use `.await`. Any iterator chain that would call an async
method inside `.filter()`, `.find()`, or `.map()` must be rewritten as a `for`
loop. This came up in `send_direct_message` (finding an existing DM room) and
`get_user_profile` (finding shared rooms).

## 17. How to Add a New Tool

1. **Define the input struct** (if the tool has parameters):

```rust
#[derive(Debug, Deserialize, schemars::JsonSchema)]
pub struct MyToolInput {
    #[serde(rename = "myParam")]
    pub my_param: String,
}
```

2. **Add a method** inside the `#[tool_router] impl MatrixMcpServer` block:

```rust
#[tool(description = "Does something useful")]
async fn my_tool(
    &self,
    Parameters(input): Parameters<MyToolInput>,
) -> Result<CallToolResult, McpError> {
    // your logic
    Ok(single_text("Done!"))
}
```

3. **Rebuild** (`cargo build`). The tool is automatically registered in the
   router. No manual wiring needed.

4. **Test with MCP Inspector**:

```bash
cargo run &
npx @modelcontextprotocol/inspector
# connect to http://localhost:3000/mcp
```
